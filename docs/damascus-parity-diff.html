Mitigation Strategy

  • app.py — each feature adds its keybinding + modal in an isolated block; merge Phase 0 first so later branches start from a shared base
  • config.py — add new constants at end of file; use section comments to reduce overlap
  • New modules — most features add new files (summarizer/, export/, sharing/) which don't conflict
  • Merge order matters — merge foundation (Phase 0) before any Phase 1/2 branches start
  • Rebase before merge — rebase feature branches onto main after each phase merge

Recommended Implementation Order

Phased rollout with dependency-aware sequencing

Phase 0 Foundation

  • Session Store — SQLite table for sessions (id, timestamps, transcript, metadata). All later features depend on this.
  • Config Store — extend .state.json with full settings (audio retention, export format, summarization strength).
  • Audio Retention — save raw PCM alongside transcript when enabled. Toggle in config.

Merge all 3 into main before proceeding.

Phase 1a Export & Edit

  • JSON export — structured output with segments, speakers, timestamps, metadata.
  • PDF export — add reportlab or weasyprint. Speaker colors, header, page breaks.
  • Inline editing — make transcript segments editable. Tap-to-edit with undo.
  • Session browser — list past sessions, open, delete. Search by date/speaker.

Phase 1b Summarization

  • Summarization engine — local LLM (MLX-compatible, e.g. Qwen2.5-3B or Llama-3.2-3B). Load/unload to manage memory with STT model.
  • Summarization UI — strength selector (low/med/high maps to system prompt variation). Progress indicator.
  • Re-transcription — replay stored audio through different STT model. Version tracking.

Phase 2 Sharing

  • Export encryption — encrypt Markdown/JSON/PDF before leaving the app. AES-256-GCM with password-derived key.
  • FileVerse — SDK integration. Upload encrypted blob, return share link.
  • CryptPad — API integration. Create encrypted pad, return URL.

Risk: FileVerse SDK maturity unknown. Prototype early.

Memory Budget Concern

  • Qwen3-ASR 0.6B ≈ 600MB VRAM
  • Qwen3-ASR 1.7B ≈ 1.5GB VRAM
  • Summarization LLM 3B ≈ 1.5-3GB VRAM
  • Strategy: Unload STT model before loading summarizer. Summarization is post-session only — no need for concurrent inference.

Risks & Open Questions

What could go wrong

High Risk

  • FileVerse SDK — does it support fully on-device encryption before upload? Needs prototype.
  • Memory pressure — STT + summarization LLM may not fit in 16GB. Requires sequential model loading.
  • app.py merge conflicts — every feature touches this file. Strict merge ordering required.

Medium Risk

  • CryptPad API — unclear if programmatic doc creation is supported or if it's link-only.
  • PDF generation — weasyprint needs system-level dependencies (pango, cairo). reportlab is pure Python but less pretty.
  • Inline editing UX — Textual's RichLog is append-only; editable transcript needs a custom widget.

Low Risk

  • JSON export — straightforward serialization of existing data model.
  • Config store — extends existing .state.json pattern.
  • Session store — SQLite already in use for speakers; add sessions table.
  • Audio retention — simple file I/O alongside existing auto-save.

Open Questions (from spec)

  • Minimum device specs for acceptable summarization perf?
  • License choice: GPLv3 vs Apache 2.0?
  • Should raw audio be retained by default or opt-in?
  • Can STT + summarization models share MLX context?
